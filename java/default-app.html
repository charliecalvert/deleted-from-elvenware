Here you will find a tutorial on creating JBuilder applications.

Let's step through these wizards quickly just to make sure that the basics are in proper focus.

First, create a new project by Choosing File | New Project from the JBuilder menu. Creating a new project means that you are doing two things:

  1. Creating a JPR/JPX file. If you are unsure which to create, choose JPX.
  2. Creating a directory to house the file.

You don't have to do anything special to create the directory and its JPR/JPX file. If you step through the wizard, JBuilder will create them both for you. In fact, the main purpose of the wizard is to create these two items in the simplest manner possible.

After creating the project, the next step is to add one or more applications or applets to the project. As a rule, you only have one application or applet in each project. However, it is not wrong, or incorrect, to put multiple applications in one project. It is simply a matter of doing what seems most reasonable and easy to maintain.

In the previous chapter, you saw several different examples of stepping through the project and application wizards. To create a default JBuilder project, you take the same steps as in the last chapter, except you almost always stick with the defaults. For the sake of clarity, I will quickly step you through each process.

Creating a New Project

Begin by choosing File | New Project from the JBuilder menu. Alternatively, you can choose File | New, and then choose Project from the second page of the Object Gallery. In either case, you will end up in the Project Wizard, the first page of which is shown in Figure 1.

Figure 1: Using the first page of the project wizard to change the default name of a project from untitledx to MyProject

There are only a few fields that you need to edit when creating a default project. In particular, you need to name the new directory in which your project will be located, and then name the JPR or JPX file you place inside it.

NOTE: Remember, a JPR file is an ordinary text file, a JPX file is a text file in XML format.

By default, JBuilder creates projects in a directory called untitledx, where X is a scalar value larger than 0. For instance, this field might be set to untitled1, or untitled2, etc. You will usually want to rename this value to something more meaningful. Figure 1 shows a project wizard altered so that the project resides not in a directory called untitledx, but in a directory called MyProject.

The second page of the project wizard allows you to change the structure of your project itself. We experimented some with these values in the previous chapter. In this case, however, we want to stick with the default values for your project's structure. As a result, no changes should be made to the second page in the project Wizard.

The third page in the project wizard allows you to edit the text that will appear in the comments JBuilder puts by default at the top of your source files. This is an entirely intuitive process, and needs no explanation from me.

NOTE: You can change the root directory in which your projects are stored. This process will be explained below in the section of this chapter called "Changing the Default Path for a Project."

Creating a New Application

After you have created a project, you will almost always want to insert an application or applet into it. To do this, begin by first selecting File | New. On the New page of the JBuilder dialog, select Application. A dialog such as that shown in Figure 2 will appear.

Figure 2: Changing the name of an application from Application1 to MyApplication on the first page of the Application Wizard.

The application or applet you add to your project will, by default, have a name like Applicationx or Appletx. Again, the value x will be an integer value larger than 0. The files created will reside by default in a package named after your project. For instance, they might be in a directory called untitledx. In our case, we are placing them in a directory called myproject, since that was the name we gave the project in the previous section.

Figure 2 shows a JBuilder application with its name changed from the default value of Application1 to MyApplication. This page also allows you to change the name of the main package for your project. On the second page of the application wizard, you can change the name for the main Frame of your project.

When you are done, press Finish to close the Application wizard. In many cases, you will be able to press Finish on the first page without stepping through the second of the two pages of this wizard.

The second page of the wizard allows you to change the name of the Frame for your application from Frame1 to some more meaningful name. Here you can also decide whether or not to automatically generate a menu bar, tool bar, status bar and about dialog, as well as whether or not you want to center your frame in the middle of the screen when it appears.

The kind of features and editorial changes discussed in this section of the text are very straightforward. Unless you are very new to tools like JBuilder, you should have no trouble making these kinds of changes, or understanding what effect these changes have on your code.

What is less obvious, however, is the meaning and function of all the files generated by the project and application wizards. That subject will be discussed in the next section of the text.

NOTE: Remember that JBuilder does not contain any black boxes. If you set the name of a directory in JBuilder, or if you set the name of a class, then you are not writing your changes in stone. Even if you exit JBuilder, you can still change the name of that directory or the name of that class exactly as you would when working with any other directory on your system or any other piece of Java source code. The JBuilder wizards are just Borland's way of helping you fill in the parts of your code that you would have to type out or create for yourself anyway. The developers of JBuilder are religious about ensuring that development in the environment involves no black boxes or deep secrets. After all, they are developers themselves, and they want exactly the same kinds of flexibility in project development that you would want.

Later in the book, in the chapter called "Refactoring and UML," you will learn how to automatically change package or class names in your project. Refactoring changes not only the name of a single directory or a single class, but also all references to that class or package throughout your project.
